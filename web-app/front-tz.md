# Техническое задание: интерфейс Auth Service для Codurity

## 1. Назначение

Необходимо реализовать пользовательский интерфейс для модуля авторизации и управления безопасностью аккаунта Codurity.

Модуль должен закрывать полный цикл работы пользователя с аккаунтом:

* регистрация;
* подтверждение email;
* вход;
* выход;
* восстановление пароля;
* смена пароля;
* смена email;
* просмотр данных текущего аккаунта;
* удаление аккаунта;
* обработка ошибок, истёкших кодов, rate limit и неавторизованных состояний.

Интерфейс должен восприниматься как часть технического продукта для найма разработчиков: строгий, быстрый, понятный, с акцентом на доверие, безопасность и ясные пользовательские сценарии.

## 1.1. Конвенции API

Все запросы к API направляются на базовый URL:

```
/api/v1
```

причем адрес бекенда локального - http://codurity.local

### Refresh token: cookie mechanism

Refresh token передаётся через httpOnly cookie:

* имя: `refresh_token`;
* path: `/api/v1/auth`;
* httpOnly: `true`;
* SameSite: `None` при HTTPS, `Lax` при HTTP;
* MaxAge: настраивается сервером (по умолчанию 30 дней).

Фронтенд **не должен** пытаться хранить refresh token в localStorage, sessionStorage или передавать вручную. Cookie управляется автоматически при запросах к `/api/v1/auth/refresh`. При logout cookie очищается сервером; для подстраховки фронт может удалить его через `document.cookie`.

### Промежуточные токены в multi-step flows

Некоторые сценарии (смена пароля, смена email) возвращают промежуточные токены, которые фронт должен хранить скрыто от пользователя и передавать на следующем шаге:

| Токен | Получаем из | Передаём в | TTL |
|---|---|---|---|
| `reset_token` | `POST /password/forgot/verify` | `POST /password/reset` | 15 мин |
| `change_token` | `POST /password/change/verify` | `POST /password/change/complete` | 15 мин |
| `identity_token` | `POST /me/email/change/verify` | `POST /me/email/change/confirm` | 15 мин |

Рекомендация: хранить эти токены в **sessionStorage** (не localStorage — чтобы не висеть между сессиями, не в URL — чтобы не утекать). Не показывать пользователю. При закрытии вкладки или истечении TTL — начинать flow заново.

### Отзыв сессий (session revocation)

При смене пароля, смене email и удалении аккаунта сервер **отзывает все активные сессии** пользователя (кроме текущей — для смены пароля и email). Фронт должен:

* показать уведомление: "Все остальные сессии были завершены";
* при удалении аккаунта — очистить auth-state и перенаправить на `/auth/account-deleted`.

### HTTP-коды ошибок

Ключевые HTTP-коды, которые фронт должен различать:

| Код | Ситуация |
|---|---|
| `400` | Malformed JSON |
| `401` | MISSING_AUTH_TOKEN, INVALID_AUTH_TOKEN, EXPIRED_AUTH_TOKEN, INVALID_CREDENTIALS, INVALID/EXPIRED_REFRESH_TOKEN |
| `403` | EMAIL_NOT_VERIFIED (при входе), INVALID_AUTH_TOKEN (identity not active/deleted) |
| `404` | NOT_FOUND |
| `409` | EMAIL_ALREADY_EXISTS |
| `422` | INVALID_CODE, CURRENT_PASSWORD_INCORRECT, RESET_TOKEN_INVALID/EXPIRED, VALIDATION_ERROR |
| `429` | TOO_MANY_ATTEMPTS, RATE_LIMITED |

Важно: `EMAIL_NOT_VERIFIED` при входе возвращает **403**, не 401. Фронт должен направлять на страницу подтверждения email, а не на повторный вход.

## 2. Пользовательские роли

### 2.1. Гость

Пользователь без активной авторизации.

Доступные действия:

* открыть страницу входа;
* создать аккаунт;
* подтвердить email после регистрации;
* восстановить пароль;
* запросить повторную отправку кода;
* перейти между страницами входа, регистрации и восстановления пароля.

### 2.2. Авторизованный пользователь

Пользователь с активной сессией.

Доступные действия:

* просмотреть текущий аккаунт;
* выйти из аккаунта;
* сменить пароль;
* сменить email;
* удалить аккаунт;
* повторно отправлять коды подтверждения в рамках разрешённых сценариев;
* видеть статус аккаунта и email verification.

### 2.3. Пользователь с неподтверждённым email

Пользователь, который зарегистрировался, но ещё не подтвердил email.

Поведение:

* при попытке входа или доступа к приватным разделам должен быть направлен на экран подтверждения email;
* должен видеть понятное объяснение, что аккаунт создан, но требует подтверждения;
* должен иметь возможность повторно отправить код.

## 3. Общие требования к UX

Интерфейс должен быть пошаговым и не перегружать пользователя техническими деталями.

Основные принципы:

* каждое действие должно иметь один главный CTA;
* формы должны быть короткими;
* ошибки должны объяснять, что произошло и что делать дальше;
* при отправке кода пользователь должен видеть, куда отправлен код;
* при повторной отправке кода должна быть защита от частых кликов через cooldown;
* опасные действия должны требовать явного подтверждения;
* все security-сценарии должны иметь success-state;
* все loading-state должны блокировать повторную отправку формы;
* формы не должны терять введённые данные при ошибке, кроме паролей и кодов, если это нужно для безопасности.

## 4. Дизайн-направление

### 4.1. Общий стиль

Интерфейс строится в стиле Codurity:

* чистый SaaS-интерфейс;
* чёрно-белая база;
* синий цвет только как action/brand accent;
* заметные border-линии;
* умеренные скругления;
* без чрезмерного brutalist-эффекта;
* без избыточной декоративности.

### 4.2. Цвета

Основной action-цвет: синий Codurity.

Использование:

* primary-кнопки;
* активные пункты меню;
* focus-ring;
* progress/stepper;
* важные badge;
* ссылки;
* ключевые состояния success-flow.

Не использовать синий как массовую заливку фона всей страницы.

### 4.3. Типографика

* Заголовки: Manrope.
* Основной текст, формы, кнопки, навигация: Onest.
* Технические значения, ID пользователя, токен-подобные значения, коды ошибок: JetBrains Mono.

Логотипный шрифт не использовать в интерфейсе.

### 4.4. Бордеры и радиусы

* Основные карточки, inputs, кнопки, таблицы — с видимым border.
* Базовый radius для кнопок и inputs — около 10px.
* Базовый radius для карточек — около 14px.
* Большие панели могут иметь radius около 22px.
* Hard-shadow использовать редко: только для главной auth-карточки или важного security-блока.

### 4.5. Темы

Нужно поддержать light theme как основную.

Dark theme допустима и желательна для:

* технических экранов;
* страниц безопасности;
* будущих code-review/editor-like разделов.

Auth-экраны по умолчанию — light theme, но должны корректно работать и в dark theme.

В dark theme логотип должен быть белым (инвертированным).

### Поля

Password:

* label: "Пароль";
* placeholder: "Введите пароль";
* обязательное;
* тип password;

### Состояния

Loading:

* кнопка "Войти" переходит в состояние "Входим…";
* поля блокируются от повторной отправки.

Success:

* пользователь перенаправляется в продукт или на `/settings/profile`.

Errors:

* invalid credentials (401): "Email или пароль указаны неверно."
* email not verified (403): показать alert и кнопку "Подтвердить email". **Важно:** API возвращает HTTP 403, не 401. Фронт должен различать эти коды и направлять на страницу подтверждения email, а не на повторный ввод пароля.
* rate limited (429): "Слишком много попыток. Попробуйте позже."
* server error (500): "Не удалось войти. Попробуйте ещё раз."

### CodeInput

Требования:

* визуально разделённые ячейки;

### Resend

* после клика disabled на время cooldown;
* показывает текст "Повторно через 00:30";
* при успехе показывает toast "Код отправлен повторно".

### Errors

* invalid code: "Код неверный."

* иконка крестика (danger-стиль);

## 11. Responsive

### Desktop

* Auth-card центрируется на странице (max-width: 472px).
* Settings layout — sidebar (220px) + content.
* Header показывает логотип, навигацию и user menu.

### Tablet

* Auth-card сохраняет максимальную ширину.
* Settings sidebar может быть уже.

### Mobile

* Auth-card занимает почти всю ширину экрана (padding уменьшается).
* Header минимальный (без навигации).
* Settings sidebar превращается в горизонтальные tabs.
* Кнопки в формах могут быть full-width.
* CodeInput ячейки уменьшаются (44x52).
* Security cards складываются вертикально.
* Modal на mobile может открываться как bottom sheet или full-screen panel.

## 12. Accessibility

Обязательные требования:

* все inputs имеют label;
* ошибки связаны с полями через доступное описание (role="alert");
* кнопки имеют понятный текст;
* icon-only buttons имеют aria-label;
* modal имеет focus trap и role="dialog" aria-modal="true";
* после ошибки фокус переводится на alert или первое ошибочное поле;
* CodeInput доступен для клавиатуры;
* цвет не является единственным способом передать ошибку или статус;
* contrast должен быть достаточным в light и dark theme.

## 13. Security UX

* Не показывать access token, refresh token, reset token, identity token и change token пользователю.
* Не раскрывать существование email на странице восстановления пароля.
* При истечении сессии показывать понятный экран повторного входа.
* Для смены email, пароля и удаления аккаунта использовать пошаговое подтверждение.
* Удаление аккаунта всегда требует danger modal и повторное подтверждение.
* После logout очищать пользовательское состояние интерфейса.
* После удаления аккаунта не оставлять доступ к приватным страницам через browser back.

## 15. Acceptance criteria

### Регистрация

* пользователь может создать аккаунт по email и паролю (без подтверждения пароля);
* после регистрации он попадает на подтверждение email;
* код можно ввести вручную или вставить;
* код можно запросить повторно (cooldown 30 сек);
* ошибки invalid/expired/rate limit отображаются понятным текстом.

### Вход

* пользователь может войти по email и паролю;
* при неверных данных видит понятную ошибку;
* при неподтверждённом email направляется на подтверждение (403);
* при успешном входе попадает в авторизованный интерфейс.

### Восстановление пароля

* пользователь может запросить код восстановления;
* UI не раскрывает, существует ли email;
* после подтверждения кода можно задать новый пароль;
* после успеха показывается success-state с уведомлением о завершении сессий.

### Профиль

* пользователь видит account card с email, статусом, датой создания и ID;
* статусы отображаются badge-компонентами;
* на профиле есть быстрые действия: "Изменить email", "Сменить пароль", "Удалить аккаунт";
* при потере авторизации пользователь перенаправляется на вход.

### Смена пароля

* пользователь подтверждает текущий пароль;
* вводит код (с cooldown повторной отправки);
* задаёт новый пароль (с индикатором требований);
* видит success-modal;
* видит уведомление "Все остальные сессии были завершены".

### Смена email

* пользователь подтверждает личность паролем;
* подтверждает код на текущий email;
* вводит новый email;
* подтверждает код на новый email;
* после успеха видит обновлённый email в профиле;
* видит уведомление "Все остальные сессии были завершены".

### Удаление аккаунта

* пользователь видит предупреждение о последствиях со списком;
* вводит пароль;
* подтверждает удаление кодом;
* после успеха выходит из аккаунта и видит страницу "Аккаунт удалён".

### Дизайн

* все страницы соответствуют Codurity Design System;
* primary actions используют синий брендовый цвет;
* формы, карточки и кнопки имеют видимые бордеры;
* интерфейс остаётся чистым и техническим;
* hard-shadow используется только на ключевых блоках;
* mobile-адаптация работает для всех auth-сценариев.

## 16. Что не входит в текущий объём

В текущий объём не включать:

* управление организациями;
* роли компании;
* приглашения пользователей;
* список активных сессий;
* двухфакторную авторизацию через приложения;
* social login;
* SSO;
* админскую блокировку аккаунтов;
* управление кандидатами, вакансиями, тестами и отчётами.

Эти функции могут быть добавлены отдельными API и отдельным ТЗ.
